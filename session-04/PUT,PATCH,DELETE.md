# 1. 리소스 수정: PUT vs PATCH
### 1.1 PUT (Full Update)
* **정의**: 리소스의 **전체 교체**를 의미합니다. 클라이언트는 수정하려는 리소스의 모든 필드를 포함하여 요청을 보내야 합니다.
* **동작 방식**: 서버에 해당 리소스가 있으면 전체를 덮어쓰고, 없으면 새로 생성합니다 (**Upsert**).
* **주의 사항**: 요청 데이터에서 특정 필드를 누락할 경우, 서버는 해당 필드를 `null`로 인식하거나 기본값으로 초기화할 수 있어 데이터 유실 위험이 있습니다.
* **특징**: **멱등성(Idempotency) 보장**. 동일한 전체 데이터를 여러 번 보내도 결과는 항상 동일합니다.

### 1.2 PATCH (Partial Update)
* **정의**: 리소스의 **일부 수정**을 의미합니다. 변경이 필요한 필드만 전송하므로 효율적입니다.
* **동작 방식**: 서버는 전달받은 필드값만 기존 리소스에 반영합니다.
* **장점**: 네트워크 페이로드 크기를 줄일 수 있어 대용량 리소스 수정 시 유리하며, 전송하지 않은 데이터는 보존됩니다.
* **특징**: 설계 방식에 따라 **비멱등성**을 가질 수 있습니다. (예: 특정 필드 값을 '증가'시키는 로직이 포함된 경우 결과가 달라짐)

| 메서드 | 의미 | 예시 (Request Body) | 비고 |
| :--- | :--- | :--- | :--- |
| **PUT** | 전체 덮어쓰기 | `{"name": "Gemini", "age": 20, "bio": "Hello"}` | 누락된 필드는 유실될 위험 있음 |
| **PATCH** | 부분 필드 변경 | `{"age": 21}` | 필요한 데이터만 전송, 효율적임 |

# 2. 리소스 삭제: DELETE
서버의 특정 리소스를 삭제하기 위해 사용합니다.

* **설계 원칙**: 삭제하려는 리소스의 식별자(ID)를 URI에 포함합니다. (예: `DELETE /users/10`)
* **반환 코드**:
    * **200 OK**: 삭제 성공 및 응답 메시지 포함.
    * **204 No Content**: 삭제 성공했으나 응답할 본문이 없음.
* **멱등성**: 이미 삭제된 리소스를 다시 DELETE 요청하더라도 서버의 리소스 상태는 '삭제됨'으로 동일하며, 클라이언트는 404를 받을지언정 서버 데이터가 추가로 변하지는 않습니다.
