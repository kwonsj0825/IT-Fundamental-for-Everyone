# HTTP 의 특성
HTTP 메세지에 아래의 모든것이 담겨 전송된다.
- HTML, TEXT
- IMAGE, 음성, 영상 파일
- JSON, XML

거의 모든 형태의 데이터가 전송 가능하며, 서버간에 데이터를 주고받을 때도 대부분 HTTP 를 사용한다.

## TCP 와 UDP 의 각 기반 프로토콜
- TCP : HTTP/1.1, HTTP/2
- UDP : HTTP/3
- HTTP/1.1 을 주로 사용하나 HTTP/2, HTTP/3 도 점점 증가하는 추세라고 한다.

## HTTP 의 특징
- 클라이언트 - 서버 구조이다.
- 무상태 프로토콜(Stateless) 이며, 비연결성의 특징을 가지고 있다.
- HTTP 메세지를 통해서 통신하며, 단순하고, 확장 가능하다.

## 클라이언트 - 서버 구조
- HTTP 통신은 클라이언트 - 서버 구조이다.
- 클라이언트에서 서버로 요청을 보내고(Request), 서버로부터 돌아오는 응답(Response) 을 클라이언트가 대기하는 방식으로 통신이 진행된다.
- 서버가 요청에 대한 결과를 만들어서 응답 메세지를 클라이언트로 전송한다.

단순해 보이지만 이와같이 클라이언트, 서버로 분리해놓은 것이 굉장히 중요하다.
이렇게 분리를 함으로서 비즈니스 로직이나 데이터 같은 것들은 모두 서버에 밀어넣고, 클라이언트는 UI와 같은 사용성에 집중한다.
이렇게 설계하면 클라이언트 분야와 서버 분야가 각각 독립적으로 발전해 나갈 수 있다.
즉, 클라이언트와 서버는 서로 각각이 맡고 있는 책임에만 집중해서 기술을 발전시켜 나가면 되는 것이다.


# Stateful VS Stateless
## Stateful (상태유지 프로토콜)
- **상태유지** 란 클라이언트와 서버 관계에서 서버가 클라이언트의 상태를 보존하는 것을 의미한다.
- 클라이언트와 서버간에 송수신을 하며 단계별 과정을 진행하는데 있어 서버에서 클라이언트가 이전 단계에서 제공한 값을 저장하고,
다음 단계에서도 저장한 상태이다.
- 대표적으로 한번 로그인을 하면 페이지를 이동해도 로그인이 풀리지 않고 계속 유지되는 것이 바로 서버가 클라이언트의 상태를 유지하고 있으니까 가능한 일이다.
- 클라이언트의 정보를 기억한다는 말은 어딘가에 정보를 저장하고 통신할 때마다 읽는다는 뜻이다.
- 이런 정보들은 일반적으로 브라우저의 쿠키(cookie) 에 저장되거나, 서버의 세션(session) 메모리에 저장되어 상태를 유지하게 된다.

### 대표적인 Stateful 프로토콜 예시
대표적인 Stateful 구조를 따르는 프로토콜로 TCP 3 way handshake 가 있다.
- 클라이언트는 서버에 SYN(접속 요청) 을 전송하고 SYN_SENT 상태가 된다.
- 서버는 SYN 요청을 받고 클라이언트에 요청을 수락하는 SYN/ACK 를 전송하고 SYN_RECEIVED 상태가 된다.
- 클라이언트는 서버에게 수락 확인으로 ACK 를 또 보내고 수신 받은 서버는 ESTABLISHED 상태가 된다.
- 서버의 세션 '상태' 가 ESTABLISHED 가 됨으로서, 서버와 클라이언트는 서로 데이터를 주고받을 수 있는 상태가 된다.

이렇게 TCP 는 세션 '상태' 에 따라 서버의 응답이 달라지게 되는 Stateful 하다고 말할 수 있다.

### Stateful 의 문제점
- 상태를 유지한다는 것은 서버에서 클라이언트의 상태 정보를 저장하고 있다는것이다. 
- 여기서 문제점은 서버가 멈추거나 여러가지 이유로 인해 서버를 사용하지 못하게되어 다른 서버를 사용해야 할 때 발생한다.
- 바로 새로운 서버에서는 이전 서버에서 가지고 있던 상태값들을 가지고 있지 않기 때문이다.
- 예시) 로그인 -> 게시판에 글 작성하기 기능 실행 -> 로그인한 서버에 문제 발생 -> 다른 서버로 대체 -> 해당 서버에 로그인 상태에 대한 정보가 없음 -> 다시 로그인 화면 출력
- 만약, 기존 서버에서 다른 서버로 이전 데이터를 모두 전달해준다면 문제가 되지 않을수도 있다.
- 또한 하나의 서버가 1만명의 클라이언트를 처리할 능력이 있다고 할 때, 그보다 더 많은 수의 클라이언트가 몰리면 이미 연결된 1만명의 클라이언트 중 일부가 빠져야 다음 클라이언트가 처리된다는 한계가 있다.
- 서버의 메모리에는 용량 한계가 존재하나 클라이언트의 상태들을 계속 들고 있으니 발생하는 현상이다.
- 따라서 현업에서는 클라이언트의 상태 데이터를 따로 캐시 서버(Redis) 에 저장하여 이용한다.

## Stateless (무상태 프로토콜)
- **무상태** 는 반대로 서버가 클라이언트의 상태를 보존하지 않는것을 의미한다.
- Stateless 구조에서 서버는 단순히 요청이 오면 응답을 보내는 역할만 수행하며, 상태 관리는 전적으로 클라이언트에게 책임이 있는것이다.
- 즉, 클라이언트와 서버간의 통신에 필요한 모든 상태 정보들은 클라이언트에서 가지고 있다가 서버와 통신할 때 데이터를 실어 보내는 것이 무상태(Stateless) 구조이다.
- 이럴경우 서버는 단순히 받아서 응답만 해주기 때문에 상태유지에 대한 부하가 현저히 줄어들게 된다. 또한 상태를 보관하지 않기 때문에 특정 서버에 문제가 생겨도 다른 서버가 이어받아서 응답해도 문제가 발생하지 않는다.
- 이러한 특징으로 인해 서버 확장이 쉽기 때문에 대량의 트래픽 발생이 예상될 시, 새로 서버를 추가하는것으로 문제를 쉽게 해결할 수 있다는 장점이 있다.
- stateful 과 달리 응답 서버가 도중에 바뀌어도 클라이언트측에 정확한 응답을 보내는데 문제가 없기 때문이다.

### 대표적인 Stateless 프로토콜 예시
대표적인 Stateless 프로토콜에는 UDP 와 HTTP 가 있다. (HTTP 통신 기본이 무상태이다.)
- 무상태에서 브라우저는 데이터를 전송할 때마다 연결되었다가 바로 연결을 끊어버린다.
- UDP 를 예를 들면 TCP 와는 달리 handshaking 과정을 통해 연결 세션을 인증하는 절차가 수행되지 않고, 세션 상태에 상관없이 그냥 무작정 보내버린다.
- 서버는 클라이언트와의 세션 정보를 저장하지 않기때문에 클라이언트가 송신한 데이터가 수신되었는지 여부를 확인하지 않으며, 클라이언트와의 세션 상태에 상관없이 요청에 대한 응답만 수행하게 된댜.
- 즉, 클라이언트와의 세션 정보를 서버가 저장하지 않는다는 점에서 Stateless 하다고 말할 수 있다.

### Stateless 의 문제점
- 클라이언트의 요청에 상대적으로 Stateful 보다 더 많은 데이터가 소모되게 된다.
- 매번 요청할 때마다 클라이언트는 자신의 부가정보를 서버측에 전달해줘야 한다.
- 물론 이벤트 소개 페이지처럼 아무 정보를 담을 필요없는 경우엔 무상태로 만들어주는것이 좋다.
- 그러나 로그인 처럼 유저의 상태를 유지해야 하는 서비스는 상태를 유지하지 않으면 로그인이 풀려버린다.
- 따라서 모든 것을 무상태로 설계할수는 없다. 어쩔수 없는 경우에만(로그인 처럼) 상태 유지를 최소한으로 사용하는 것이 베스트이다.

### Stateless 와 토큰(Token)
- 무상태의 특징은 필요한 상태 정보들을 모두 클라이언트에서 가지고 있다가 서버와 통신할 때 데이터를 실어 보내는것 이라고 했다.
- 그래서 서버는 단순히 요청을 받아서 응답만 해주면 되기 때문에 부하가 현저히 줄어든다.
- 그러나 로그인 유지와 같은 경우엔 결국 stateful 구조를 사용해야 하는데, 그러면 결국 서버에 부하가 생길수밖에 없다.
- 그 대안으로서 사용할 수 있는것이 바로 JWT(Json Web Token) 이다 JWT 는 stateless 특징을 유지하면서도 로그인 상태 유지를 가능하게 해주는 기술이다.
- 토큰은 클라이언트가 암호화된 로그인 정보들을 가지고 있다가 서버에 통신할 때 넘겨줌으로서 클라이언트가 로그인됐음을 인증하는 방식이다.
